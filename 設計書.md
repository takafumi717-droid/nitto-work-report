# 株式会社日東工業所 作業日報Webアプリ 設計書

## 1. 概要

### 1.1 目的
工場作業員がスマートフォンから簡単に作業日報を入力できるWebアプリケーションを構築する。
管理者は全従業員の日報を確認・修正・集計・CSV出力できる。

### 1.2 技術構成
| 項目 | 内容 |
|---|---|
| フロントエンド | Next.js (App Router) + TypeScript |
| スタイリング | Tailwind CSS |
| バックエンド/DB | Supabase (PostgreSQL) |
| 認証 | 一般社員: ログイン不要 / 管理者: 4桁PINによる簡易認証 |
| 対応デバイス | スマートフォン優先（レスポンシブ対応） |
| 言語 | 日本語表示のみ |

### 1.3 想定ユーザー
- **一般社員**: ログイン不要。氏名を選んで日報を入力するだけ。
- **管理者** (大久保 喬史 / 大久保 圭 / 末継 良治): PIN認証後、管理ダッシュボードを利用。

## 2. 画面構成

```
/                      … トップ画面（氏名選択）
/report/new            … 日報入力画面
/report/done           … 登録完了画面
/admin/login           … 管理者PINログイン画面
/admin                 … 管理者ダッシュボード
/admin/reports         … 日報一覧画面
/admin/reports/[id]    … 日報編集画面
/admin/summary         … 月次集計画面
/admin/csv             … CSV出力画面
/admin/employees       … 従業員管理画面
```

### 2.1 トップ画面 (`/`)
- 大きな見出し「作業日報」
- 従業員名を大きなボタンリストで表示（sort_order昇順、is_active=trueのみ）
- ボタンをタップ → 日報入力画面へ遷移（氏名を選択状態で渡す）
- 右下/下部に小さく「管理者ログイン」リンク

### 2.2 日報入力画面 (`/report/new`)
入力項目（最小限）:
1. 氏名（選択済み・変更可能なドロップダウン）
2. 作業日（日付ピッカー、初期値=本日）
3. 作業伝票番号（テキスト/数値入力）
4. 作業時間（0.25H刻みのセレクト or ステッパー、例: 0.25, 0.5, 0.75, 1.0 …）
5. 備考（任意、テキストエリア）

- 「登録する」大ボタンを画面下部に固定表示
- 入力チェック: 氏名・作業日・伝票番号・作業時間は必須

### 2.3 登録完了画面 (`/report/done`)
- 「登録しました」表示
- 「もう一件入力する」「トップに戻る」ボタン

### 2.4 管理者PINログイン画面 (`/admin/login`)
- 4桁のPIN入力（テンキー風UI）
- PIN一致する管理者を特定 → セッション/Cookieに管理者IDを保存

### 2.5 管理者ダッシュボード (`/admin`)
- メニュー形式（カード/ボタン）:
  - 日報一覧
  - 月次集計
  - CSV出力
  - 従業員管理
- ログアウトボタン

### 2.6 日報一覧画面 (`/admin/reports`)
- フィルタ: 対象月、従業員
- テーブル表示（日付・氏名・伝票番号・作業時間・備考）
- 各行に「編集」ボタン

### 2.7 日報編集画面 (`/admin/reports/[id]`)
- 入力項目を編集可能なフォームで表示
- 「保存」時に `report_edits` に変更前/変更後を記録
- 「削除」は行わない（要件上、修正のみ）

### 2.8 月次集計画面 (`/admin/summary`)
- 対象月を選択
- 従業員ごとの合計作業時間、日報件数を集計表示

### 2.9 CSV出力画面 (`/admin/csv`)
- 期間（開始月～終了月 or 単月）選択
- 「CSV出力」ボタン → ダウンロード

### 2.10 従業員管理画面 (`/admin/employees`)
- 従業員一覧（sort_order順）
- 追加 / 編集（氏名・PIN・管理者権限）
- 表示順変更（上下移動 or 並び替え）
- 在籍／退職切替（is_active トグル）

### 3.2 work_reports（作業日報）

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, default gen_random_uuid() | 日報ID |
| employee_id | uuid | NOT NULL, FK → employees.id | 従業員ID |
| work_date | date | NOT NULL | 作業日 |
| slip_no | text | NOT NULL | 作業伝票番号 |
| work_hours | numeric(5,2) | NOT NULL | 作業時間（0.25刻み） |
| remarks | text | NULL | 備考 |
| created_at | timestamptz | NOT NULL default now() | 作成日時 |
| updated_at | timestamptz | NOT NULL default now() | 更新日時 |

- `work_hours` には CHECK制約 `work_hours % 0.25 = 0` を付与し0.25刻みを保証
- 従業員削除は行わないため、退職者の日報も `employee_id` で参照可能

### 3.3 report_edits（修正履歴）

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, default gen_random_uuid() | 履歴ID |
| work_report_id | uuid | NOT NULL, FK → work_reports.id | 対象日報ID |
| edited_by | uuid | NOT NULL, FK → employees.id | 編集した管理者ID |
| before_data | jsonb | NOT NULL | 修正前データ |
| after_data | jsonb | NOT NULL | 修正後データ |
| created_at | timestamptz | NOT NULL default now() | 修正日時 |

### 3.4 ER概要
```
employees (1) ──< (N) work_reports (1) ──< (N) report_edits
employees (1) ──< (N) report_edits (edited_by)
```

### 3.5 RLS（Row Level Security）方針
- 一般社員はログインしないため、`work_reports` への INSERT は匿名キー（anon）で許可するが、`employee_id` が `employees.is_active=true` であることをアプリ側でチェック。
- 管理者操作（UPDATE/SELECT全件/従業員管理）はサーバーサイド（Next.js Route Handler + Service Role Key）経由で実施し、PINで認証されたセッション（Cookie）を確認する。
- Supabaseの匿名キーをフロントに置く場合は、INSERTのみ許可しUPDATE/DELETEは禁止するRLSポリシーを設定する。

## 4. 認証方式

- **一般社員**: 認証なし。トップ画面で氏名選択 → そのまま入力。
- **管理者**: `/admin/login` で4桁PINを入力 → サーバー側で `employees` テーブルの `admin_pin` と一致する `is_admin=true` レコードを検索 → 一致したら HTTP Only Cookie（暗号化済みセッション or JWT）を発行。
- 管理者用APIは Cookie のセッションを確認するミドルウェアで保護。

## 7. セットアップ手順

1. Supabaseプロジェクトを作成
2. SQL Editorで `supabase/schema.sql` を実行（テーブル作成・初期データ投入）
3. SQL Editorで `supabase/policies_anon.sql` を実行（anonキーからの一般社員向けアクセスを許可するRLSポリシー）
4. `.env.local.example` を `.env.local` にコピーし、以下を設定
   - `NEXT_PUBLIC_SUPABASE_URL`: SupabaseプロジェクトのURL（Settings > API > Project URL）
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`: anon publicキー（Settings > API）
   - `SUPABASE_SERVICE_ROLE_KEY`: Service Role キー（Settings > API、管理者画面用）
   - `SESSION_SECRET`: 任意のランダムな長い文字列（管理者セッション署名用）
5. `npm install`
6. `npm run dev` で起動し `http://localhost:3000` を確認

### 7.1 接続方式
- **一般社員向け画面**（`/`, `/report/new`, `/api/reports`）: `NEXT_PUBLIC_SUPABASE_ANON_KEY` を使用。
  `supabase/policies_anon.sql` のRLSポリシーにより、`is_active=true` かつ `is_admin=false` の従業員のみ参照・登録可能。
- **管理者画面**（`/admin/**`）: `SUPABASE_SERVICE_ROLE_KEY` を使用（RLSをバイパスし全データにアクセス）。

## 9. Vercelへの公開手順

### 9.1 GitHubへのアップロード
1. GitHub上に新しいリポジトリを作成（Public/Privateいずれでも可）
2. ローカルリポジトリは作成・コミット済み。リモートを追加してpush:
   ```bash
   git remote add origin https://github.com/<ユーザー名>/<リポジトリ名>.git
   git branch -M main
   git push -u origin main
   ```
3. `.env.local` は `.gitignore` で除外されているため、GitHubには上がらない（確認済み）。

### 9.2 Vercelでのプロジェクト作成
1. [Vercel](https://vercel.com/) にログインし、「Add New > Project」
2. 上記でpushしたGitHubリポジトリを選択してImport
3. Framework Presetは自動で「Next.js」が選択される（変更不要）
4. 「Environment Variables」セクションで、9.3の環境変数をすべて設定
5. 「Deploy」をクリック

### 9.3 Vercelに設定する環境変数

| 変数名 | 値 | 補足 |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://afmdtnbqxdpsoovansrm.supabase.co` | SupabaseのProject URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `sb_publishable_41OMc27tRvWSDDdcd66gpQ_BKBCB15M` | Supabaseの Publishable (anon) key |
| `SUPABASE_SERVICE_ROLE_KEY` | （Supabase Settings > API Keys から取得） | 管理者画面（`/admin/**`）で使用。Secretキーのため絶対に公開しない |
| `SESSION_SECRET` | `5db72e5248c09979ee4a202c87b6672a2bf414692f7b4982173d52eaee13b815`（例） | 管理者セッション署名用。本番では別の値を生成して使用すること |

**注意**:
- `SUPABASE_SERVICE_ROLE_KEY` はSupabaseダッシュボードの Settings > API > Project API keys の `service_role` キー（secret）を使用する。
- `SESSION_SECRET` は上記はサンプル値。本番用には別途ランダムな文字列を生成して設定することを推奨（例: `openssl rand -hex 32`）。
- `NEXT_PUBLIC_` で始まる変数はブラウザに公開されるため、anon/publishableキーのみを設定すること（service roleキーは絶対に `NEXT_PUBLIC_` を付けない）。
